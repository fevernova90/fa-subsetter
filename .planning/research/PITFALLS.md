# Pitfalls Research

**Domain:** Legacy CRA dependency security upgrade (react-scripts 3→5, node-sass→sass, bulk npm updates)
**Researched:** 2026-03-24
**Confidence:** HIGH (multiple corroborating sources across all major pitfalls)

---

## Critical Pitfalls

### Pitfall 1: Webpack 5 Removes Node Core Module Polyfills

**What goes wrong:**
react-scripts 5 ships Webpack 5. Webpack 4 (used in react-scripts 3) automatically polyfilled Node.js core modules (`buffer`, `crypto`, `stream`, `path`, `fs`, etc.) in the browser bundle. Webpack 5 does not. Any dependency that transitively uses a Node core module will fail to compile with the error: `BREAKING CHANGE: webpack < 5 used to include polyfills for node.js core modules by default`.

**Why it happens:**
Developers assume the upgrade is drop-in because package.json only changes one version number. The breakage is deep in transitive dependencies, invisible until the first build attempt. `fontawesome-subset` and `archiver` both run server-side in this project, but if any client-side import path reaches a Node-only module, the build will fail.

**How to avoid:**
Before upgrading react-scripts, audit all client-side imports for any library that touches Node APIs. Run `npm ls` to inspect the dependency tree. If polyfills are needed, use `react-app-rewired` with a `config-overrides.js` to add `resolve.fallback` entries — the project already has `react-app-rewired` and `customize-cra` installed as devDependencies, so this path is available without additional tooling.

**Warning signs:**
- Build fails immediately after `react-scripts` version bump
- Error messages mention `Can't resolve 'buffer'`, `Can't resolve 'crypto'`, `Can't resolve 'stream'`
- Error header says `BREAKING CHANGE: webpack < 5 used to include polyfills`

**Phase to address:**
Phase that performs the react-scripts 3→5 upgrade — must include a build smoke test immediately after version bump before any other changes.

### Pitfall 2: Existing Smoke Test Breaks on react-scripts 5 / Jest 27

**What goes wrong:**
The existing test (`App.test.js`) uses `ReactDOM.render()` directly and the old CRA smoke test pattern. react-scripts 5 ships Jest 27, which changed the default test environment from `jsdom` to `node`. Additionally, `ReactDOM.render` is deprecated in React 18 (if React is upgraded alongside react-scripts) and triggers console errors that can fail tests configured with `--ci` flag.

**Why it happens:**
The test was generated by CRA 3 scaffolding. It uses `ReactDOM.render` + `ReactDOM.unmountComponentAtNode`, the pre-React-18 API. Jest 27's changed default environment means DOM globals (`document`, `window`) may not be available without explicit `@jest-environment jsdom` annotation.

**How to avoid:**
Before upgrading react-scripts, update the test to use `@testing-library/react` (`render` + `screen`) and add `/* @jest-environment jsdom */` as a file-level docblock if needed. Do not upgrade React itself to 18 unless explicitly in scope — the project constraint is security maintenance only. Keep React at 16.x to avoid ReactDOM.render deprecation becoming an actual error.

**Warning signs:**
- `document is not defined` error in tests after upgrade
- `ReactDOM.render is no longer supported` console error in test output
- Test passes locally but fails in CI with `--ci` flag (errors-as-failures)
- Zero tests run (Jest silently skips all files due to environment mismatch)

**Phase to address:**
Baseline testing phase — write/fix tests BEFORE upgrading react-scripts, as specified in project constraints.

### Pitfall 3: Using `--force` or `npm audit fix --force` on Transitive Vulnerabilities

**What goes wrong:**
`npm audit fix --force` resolves semver beyond stated ranges, which can silently install major version bumps of direct dependencies. On a project with 239 vulnerabilities the vast majority of which are in react-scripts transitive deps, running `--force` risks replacing `react-scripts@3` with `react-scripts@5` while simultaneously force-bumping unrelated packages like `express`, `archiver`, or `redux`, each of which may have their own breaking changes.

**Why it happens:**
The output of `npm audit` makes it appear that running `npm audit fix` will handle everything. The `--force` flag is suggested in npm's own output when automatic fix is blocked. Developers interpret it as "try harder" rather than "bypass compatibility constraints".

**How to avoid:**
Never run `npm audit fix --force` on this project. Instead:
1. Upgrade react-scripts explicitly and intentionally (resolves ~200 of 239 vulns)
2. Use `npm overrides` in package.json to pin vulnerable transitive dependencies to patched versions without forcing parent upgrades
3. Upgrade each remaining direct dependency individually and verify after each one

**Warning signs:**
- `npm audit fix` output says "X vulnerabilities require manual review" or "requires `--force`"
- Post-fix `package.json` shows unexpected major version changes in unrelated packages
- App fails to start after `npm audit fix` run

**Phase to address:**
Every upgrade phase — establish a rule: no `--force` flag, verify `npm audit` output after each individual change.

### Pitfall 4: Sass Slash Division Deprecation Floods Build Output (and Hides Real Errors)

**What goes wrong:**
Dart Sass (the `sass` package) introduced a deprecation warning for `/` used as a division operator (as opposed to CSS list separator). Switching from `node-sass` to `sass` causes the build to emit hundreds of `DEPRECATION WARNING: Using / for division is deprecated` lines — most from third-party SCSS (e.g., any Bootstrap or Material-UI SCSS imports, if present). This makes it hard to distinguish genuine breaking errors from noise.

**Why it happens:**
The division deprecation has been present in Dart Sass since v1.33.0 and the syntax will be fully removed in Dart Sass 2.0.0. Projects that previously used `node-sass` never saw these warnings because node-sass is LibSass-based and does not emit them.

**How to avoid:**
After switching to `sass`, first run a build and categorize the output:
- Warnings from `node_modules` paths → use `sassOptions: { quietDeps: true }` in CRA's `SASS_PATH` or via `react-app-rewired` config to suppress third-party noise
- Warnings from `src/` paths → fix own code using `math.div()` from `sass:math`, or use `calc()` for CSS division

This project uses plain CSS files (`.css`, not `.scss`) in `src/`, so the slash-division issue is unlikely to appear in first-party code. The risk is primarily from `@fortawesome/fontawesome-free` SCSS if imported. Verify the scope before muting warnings globally.

**Warning signs:**
- Build succeeds but terminal is flooded with `DEPRECATION WARNING` lines
- Genuine compile errors are buried below hundreds of warning lines
- CI build logs become unreadable

**Phase to address:**
node-sass → sass replacement phase — run a build immediately after the swap and triage warning output before proceeding.

### Pitfall 5: Lockfile Version Mismatch Corrupts Resolution After Upgrade

**What goes wrong:**
The `package-lock.json` was generated by an older npm version (lockfileVersion 1, for npm 5-6 era). Upgrading react-scripts from 3→5 changes the entire webpack/babel/jest dependency subtree. Running `npm install` on the old lockfile with a newer npm version can produce a corrupted or inconsistent lockfile that passes `npm install` but has mismatched peer resolutions that only surface at runtime or build time.

**Why it happens:**
npm lockfile format has changed across versions (lockfileVersion 1, 2, 3). When the lockfile was created with npm 5/6 and installed with npm 8+, npm silently regenerates parts of the tree using its newer resolution algorithm, which may resolve peer deps differently than intended.

**How to avoid:**
Before starting upgrades, check the lockfile version: `cat package-lock.json | head -5`. If lockfileVersion is 1, delete `package-lock.json` and `node_modules` and do a clean `npm install` to regenerate the lockfile under the current npm version — before making any version changes. This establishes a clean baseline. Do this as the very first step of any phase that touches dependencies.

**Warning signs:**
- `npm WARN old lockfile` appears during install
- `npm install` succeeds but `npm audit` reports more vulnerabilities than expected
- `npm ls` shows `UNMET PEER DEPENDENCY` warnings after install
- Build succeeds but runtime behavior is wrong (wrong version of a package loaded)

**Phase to address:**
Pre-upgrade baseline phase — regenerate lockfile from scratch before any dependency version changes begin.

### Pitfall 6: react-app-rewired / customize-cra Version Incompatibility with react-scripts 5

**What goes wrong:**
This project already has `react-app-rewired@2.1.5` and `customize-cra@0.9.1` installed as devDependencies. These versions were written for CRA 3/4 (Webpack 4). After upgrading to react-scripts 5 (Webpack 5), existing `config-overrides.js` configurations may silently fail, apply partial config, or cause the build to hang with no error message.

**Why it happens:**
`customize-cra@0.9.x` uses internal CRA/Webpack APIs that changed between Webpack 4 and 5. The library was not updated to support Webpack 5 at the 0.9.x release line. `react-app-rewired@2.1.5` itself is more stable but `customize-cra` helpers like `addBabelPlugin`, `addWebpackPlugin`, and `adjustStyleLoaders` use internal paths that shifted.

**How to avoid:**
If no `config-overrides.js` currently exists (the project may not be using these at all — check the scripts in `package.json`), remove `react-app-rewired` and `customize-cra` as dead weight to reduce attack surface and future confusion. If a `config-overrides.js` is needed for Webpack 5 polyfill fixes, upgrade `react-app-rewired` to latest (`^2.2.x`) and use `customize-cra@1.0.0` (the Webpack 5 compatible release).

Note: The project's `package.json` scripts use `react-scripts` directly (not `react-app-rewired`), confirming that `react-app-rewired` is installed but not currently wired up. This is a cleanup opportunity.

**Warning signs:**
- `config-overrides.js` exists in project root but is not referenced by `package.json` scripts
- Build silently uses default webpack config despite overrides being defined
- Build hangs at webpack compilation step with no error

**Phase to address:**
react-scripts upgrade phase — audit and either remove or upgrade these devDependencies before upgrading react-scripts itself.

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Using `npm overrides` to pin transitive deps instead of upgrading parent | Resolves audit warnings quickly | Masks real incompatibilities; must be maintained manually forever | Acceptable for short-lived patches on abandoned upstream packages only |
| Skipping test baseline before upgrades | Faster to start | No regression detection; "it still works" is unverifiable | Never — project constraints explicitly require tests first |
| Upgrading react-scripts + React version simultaneously | One npm install | Two sets of breaking changes compound; impossible to isolate which caused failures | Never in this context — upgrade react-scripts only, keep React at 16.x |
| Silencing all Sass deprecation warnings globally (`quietDeps: true` for all) | Clean build output | Hides genuine first-party deprecations that will become errors in Dart Sass 2.0 | Only for `node_modules` scope, never for `src/` scope |
| Running `npm install --legacy-peer-deps` as default flag | Resolves install errors | Peer dependency mismatches are deferred to runtime; may load duplicate React instances | Only as last resort after exhausting proper resolution |

## Integration Gotchas

Common mistakes when connecting to external services.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| react-scripts (CRA) + react-app-rewired | Assuming `config-overrides.js` is automatically applied; not updating package.json scripts to use `react-app-rewired` binary | Replace `react-scripts` with `react-app-rewired` in all package.json script entries when using overrides |
| npm overrides field | Adding an override for a package version that is not actually semver-compatible with the parent | Always verify the overridden version's changelog for breaking changes before pinning; test build + runtime after adding each override |
| node-sass → sass | Assuming drop-in replacement with identical behavior on all sass features | Run full build and diff output; check for `@use` vs `@import` semantics and slash-division warnings |
| fontawesome-subset (server-side) | Importing it in a client-side React component (causes Webpack 5 Node polyfill errors) | Keep all `fontawesome-subset` usage strictly in the Express server code; never import in `src/` |

## Security Mistakes

Domain-specific security issues beyond general web security.

| Mistake | Risk | Prevention |
|---------|------|------------|
| Treating `npm audit` as resolved once count drops to zero | Transitive overrides can mask vulnerabilities that are still exploitable even if not reported | Verify with `npm audit --json` and check that resolved packages actually have patched versions, not just suppressed reports |
| Upgrading react-scripts but leaving `archiver@3.x` on a vulnerable version | `archiver` handles ZIP creation server-side; a vulnerability there affects the file the user downloads | Check `npm audit` output for `archiver` specifically after react-scripts upgrade; upgrade to `archiver@5.x` or later which has active maintenance |
| Leaving `express@4.17.1` unpatched on critical vulnerabilities | Express handles all server routes including file download; unpatched Express has known RCE or path traversal issues in some versions | Verify current Express version against advisories and upgrade to latest `4.x` patch if needed |
| Adding `npm overrides` for security patches without testing | Overridden sub-dependency version may be incompatible with the parent package in subtle ways | Run full test suite and manual smoke test of the file generation + download flow after each override addition |

## "Looks Done But Isn't" Checklist

Things that appear complete but are missing critical pieces.

- [ ] **react-scripts upgraded:** Verify `npm run build` succeeds, not just `npm install` — the install completing does not confirm the build works with Webpack 5
- [ ] **node-sass replaced:** Verify `npm run build` produces the same CSS output — a successful install of `sass` does not confirm the compiled styles are identical
- [ ] **npm audit at zero:** Verify `npm audit --json` contains no `critical` or `high` findings — `npm audit` exit code 0 can be achieved by overrides that suppress reporting without actually patching
- [ ] **Tests pass after upgrade:** Run `npm test -- --watchAll=false` and confirm no test failures or skipped tests — `npm test` in interactive mode can appear to pass while tests are actually pending
- [ ] **App runs after upgrade:** Start the dev server with `npm run dev` and manually trigger a subset generation + download — the build succeeding does not confirm the Express server integration still works
- [ ] **react-app-rewired resolved:** Confirm the package is either removed (if unused) or properly upgraded and wired into scripts — having it installed but non-functional is a confusing false state

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Webpack 5 polyfill build failure | LOW | Add `resolve.fallback` entries in `config-overrides.js` via react-app-rewired; or identify and remove the client-side import path reaching the Node module |
| Jest environment mismatch breaks tests | LOW | Add `/* @jest-environment jsdom */` docblock to test file; or add `"testEnvironment": "jsdom"` to jest config in package.json |
| `--force` caused unexpected major version bump | MEDIUM | `git checkout package.json package-lock.json`, delete `node_modules`, reinstall, then upgrade each dependency individually |
| Lockfile corruption causing peer dep errors | LOW-MEDIUM | Delete `package-lock.json` and `node_modules`, run clean `npm install`, verify with `npm ls` for UNMET PEER DEPENDENCY warnings |
| Sass slash-division warnings flood CI | LOW | Add `sassOptions: { quietDeps: true }` via CRA environment or react-app-rewired to suppress `node_modules` warnings only |
| react-app-rewired incompatibility with Webpack 5 | MEDIUM | Upgrade to `react-app-rewired@2.2.x` + `customize-cra@1.0.0`; or remove both and use only standard react-scripts if no overrides are needed |

## Pitfall-to-Phase Mapping

How roadmap phases should address these pitfalls.

| Pitfall | Prevention Phase | Verification |
|---------|------------------|--------------|
| Webpack 5 Node polyfill removal | react-scripts upgrade phase | `npm run build` succeeds without polyfill errors |

| `npm audit fix --force` accidental breakage | Every upgrade phase (process rule) | git diff after each change shows only intended version bumps |
